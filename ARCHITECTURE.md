# EquationAI — Architectural Reasoning

*Why this project is built the way it is, and what I learned building it.*

## 1. Core thesis: why neuro-symbolic, not "just ask the LLM"

The obvious way to build a math-solving app is to send the problem straight to an LLM and display whatever it says. I didn't do that, because LLMs are fundamentally unreliable at arithmetic and symbolic manipulation — they predict plausible-sounding tokens, not verified calculations. A model can sound completely confident while being numerically wrong.

Instead, EquationAI splits the problem into two stages that play to different strengths:

- **The LLM (Gemini) only handles language.** It reads the natural-language problem and decomposes it into a structured, step-by-step JSON plan — defining variables, forming equations, deciding what operation each step represents.
- **SymPy handles all actual computation.** Every step's math is executed by a deterministic symbolic engine, not generated by the model.

This means the final numeric or symbolic answer a user sees was never "written" by the LLM — it was computed. The LLM's only job is figuring out *what* to compute, not *what the answer is*. That single design decision is what eliminates hallucination at the source, rather than trying to prompt the model into being more careful.

## 2. Sandboxing untrusted, LLM-generated code

Since the LLM's output is Python-like code that gets executed via `exec()`, that output has to be treated as **untrusted input** — not because the LLM is malicious, but because any system that runs generated code needs to assume it could go wrong, whether through a bug, an unexpected edge case, or a future prompt-injection style attack.

The executor restricts `__builtins__` down to nothing, then injects only SymPy's own function names into the execution namespace. Concretely, this means LLM-generated code can call `solve()`, `integrate()`, `symbols()` — but has no path to `__import__`, `open()`, or any other builtin that could touch the filesystem or run arbitrary commands. This is a security decision, not a readability one: it makes an entire class of failure (malicious or accidental system access) structurally impossible, regardless of what the model generates.

## 3. The mismatch verifier: catching hallucination directly

Splitting parsing from computation removes hallucination from the *final answer's arithmetic* — but it doesn't automatically catch every possible failure mode. To close that gap, the system prompt requires the LLM to independently state what it believes the final answer is (`expected_final_answer_sympy`), completely separate from the step-by-step execution.

After SymPy executes the full plan, a verifier compares the LLM's stated answer against the engine's actual computed result. If they disagree, the system flags it rather than silently trusting either side. This is a direct, deterministic hallucination check: it doesn't rely on the LLM being self-aware about its own uncertainty — it mechanically checks two independently-produced values against each other.

Building this out surfaced a real design subtlety: comparing symbolic answers isn't just string equality. Two mathematically identical expressions (`pi*log(2)/8` vs a rounded decimal `0.2722`) can look completely different as strings. The verifier tries numeric comparison first (with a small tolerance, to handle legitimate rounding from numerical fallbacks), and falls back to symbolic simplification (`sympy.simplify(a - b) == 0`) when the expression contains free variables that can't be reduced to a plain number — like general solutions to differential equations or line equations.

## 4. Why systematic evaluation, not spot-checking

Testing a handful of problems by hand tells you a system worked in the cases you happened to try — it doesn't tell you whether your system prompt actually generalizes across problem types, or where it quietly breaks. I built an automated, resumable evaluation harness specifically to answer that question honestly rather than assume it.

The suite covers 22 problems across 14 categories: arithmetic word problems, unit conversion, systems of equations, standard and hard integration (including non-obvious substitution tricks), differential equations, probability, combinatorics, coordinate geometry, series, linear algebra, and deliberately ambiguous or nonsensical inputs designed to test whether the system correctly recognizes when *not* to answer.

This surfaced real gaps that spot-checking would have missed entirely — for example, the system prompt initially had no explicit guidance for differential equations, causing `dsolve()` to fail whenever the LLM defined the dependent variable as a plain symbol instead of an undefined function. That's exactly the kind of category-specific failure that only shows up with structured, categorized testing, not ad hoc examples.

The suite also distinguishes three outcomes, not two: auto-verified pass, genuine failure, and "solvable but has multiple mathematically equivalent valid forms" (needs manual review) — because a naive pass/fail scorer would incorrectly flag a correct-but-differently-formatted answer as wrong.

**Final result: 22/22 — 19 auto-verified passes, 3 manually confirmed correct.**

## 5. Case study: two real bugs, and what they taught me

**Bug 1 — the undefined-symbol failure.** Partway through development, problems started intermittently failing with `name 'x' is not defined`. The root cause: when I updated the system prompt to add a new field, I accidentally removed the concrete example steps that had been implicitly teaching the LLM to always define symbols via explicit assignment (`x = symbols('x')`) before using them. Without that anchor, the LLM sometimes generated a bare expression (`symbols('x')`, no assignment) — valid Python, but it never actually stored `x` anywhere. The fix was restoring explicit examples and adding an explicit rule: *every symbol must be created via assignment, never assumed*. The lesson: system prompt examples aren't decoration — removing them silently degrades behavior in ways that only show up intermittently, which makes them easy to miss and important to test for deliberately.

**Bug 2 — the numeric-vs-symbolic verifier gap.** The verifier initially only handled numeric comparison (`complex(sympy.N(value))`). This worked for months of testing because every test happened to produce a plain number. It broke the moment I added differential equations and line-equation problems, whose correct answers legitimately contain free variables (`C1*exp(2*x) + C2*exp(3*x)`, `y = 2x - 1`) that can never be reduced to a number — `complex()` has no way to convert an expression containing an unbound variable. The fix was a two-path comparison: try numeric equality first, and fall back to symbolic simplification when the expression contains free symbols. The lesson: a verifier (or any test) that's only ever been exercised against one *shape* of answer will silently have blind spots for other valid shapes — expanding test coverage is what actually reveals those blind spots, not more testing of the same kind.

## 6. Tech stack

- **Frontend:** Next.js 16 (React 19), Tailwind CSS, KaTeX for LaTeX rendering
- **Backend:** FastAPI, Python
- **LLM:** Google Gemini (`gemini-2.5-flash`), via the current `google.genai` SDK (migrated proactively from the deprecated `google.generativeai` package)
- **Computation engine:** SymPy, sandboxed execution
- **Evaluation:** custom resumable test harness, 22 problems across 14 categories, JSON-logged results

## 7. What I'd improve next

- Deploy publicly (currently runs locally)
- Expand evaluation coverage further (3D geometry, complex numbers, multi-part problems)
- Add unit tests for the FastAPI endpoints and error paths directly, not just end-to-end evaluation
- Add basic rate-limiting/auth before any public deployment with real user traffic
